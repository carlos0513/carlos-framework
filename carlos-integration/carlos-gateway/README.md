# carlos-gateway

钉钉、微信等平台采用随机码形式（Opaque Token）的access_token而非JWT格式，主要基于以下设计考量和技术决策：

---

一、核心机制差异

1. 信息承载方式
    - 随机码Token：仅作为凭证标识，不包含业务信息，需通过认证中心接口查询关联数据（如用户权限、有效期等）
    - JWT：自包含结构化数据，可直接解析出用户身份、权限等字段，无需额外查询

2. 验证逻辑
    - 随机码需通过中心化验证（如调用`/check_token`接口），服务端可实时控制有效性
    - JWT依赖本地签名验证，需预置公钥且难以实现即时吊销

---

二、平台级设计的核心原因

1. 安全控制强化
    - 随机码支持更细粒度的权限回收（如单用户/设备级吊销），而JWT需依赖黑名单机制或缩短有效期
    - 防止敏感信息泄露：微信/钉钉等生态涉及多企业数据隔离，JWT携带用户信息可能违反最小化暴露原则

2. 系统架构适配
    - 高并发场景优化：千万级QPS下，中心化Token验证可通过缓存+集群扩展，而JWT签名验证消耗更多CPU资源
    - 兼容历史系统：微信早期使用OAuth2协议设计，延续随机码模式可避免架构重构

3. 生态管控需求
    - 第三方应用必须通过平台接口获取用户数据，避免直接解析JWT导致数据滥用
    - 统一审计能力：所有Token使用记录可通过中心化日志追踪

---

三、技术对比总结
| 特性 | 随机码Token | JWT |
|---------------------|--------------------------------|----------------------------------|
| 信息透明度 | 不透明，需查询认证中心 | 自包含，可直接解析 |
| 吊销时效性 | 实时生效 | 依赖黑名单或短有效期 |
| 网络依赖 | 需要调用认证接口 | 本地验证，无网络依赖 |
| 适用场景 | 中心化高安全系统 | 分布式无状态架构 |

---

四、典型场景验证

1. 微信Token设计
    - 普通access_token每日限2000次获取，强制开发者缓存使用
    - 网页授权token需通过OAuth2 code交换，全程中心化验证

2. 钉钉Token异常处理
    - 出现"access_token is blank"等错误时，需重新走完整OAuth流程

---

五、扩展思考：JWT的潜在适用场景
若平台需支持以下特性时可能引入JWT：

- 离线业务处理（如物联网设备指令）
- 跨域单点登录（SSO）
- 微服务间低延迟认证

但需配套密钥轮换、短期有效期等安全措施

---

通过这种设计选择，微信/钉钉等平台在安全性、生态管控和系统扩展性之间取得了平衡，而JWT更适用于对无状态性和解析效率要求更高的场景。
