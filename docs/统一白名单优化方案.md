# Carlos Gateway 统一白名单优化方案

> **优化日期**: 2026-04-10  
> **目标**: 解决多个过滤器重复检查白名单的性能问题

---

## 一、问题分析

### 1.1 重复检查现象

在原始实现中，多个过滤器都独立进行白名单检查：

```
请求到达
    ↓
┌─────────────────────────────────────────────────────────────┐
│ PathPrefixFilter                                            │
│   - 检查白名单: PathMatchUtil.antMatchAny(whitelist, path)  │
│   - O(N) 复杂度，N = 白名单数量                             │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ OAuth2AuthenticationFilter                                  │
│   - 检查白名单: PathMatchUtil.antMatchAny(whitelist, path)  │
│   - 再次执行 O(N) 匹配                                      │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ OAuth2AuthorizationFilter                                   │
│   - 检查白名单: PathMatchUtil.antMatchAny(whitelist, path)  │
│   - 第三次执行 O(N) 匹配                                    │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│ ReplayProtectionFilter                                      │
│   - 检查白名单: startsWith 检查                             │
│   - 第四次执行 O(N) 匹配                                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 性能损耗

假设有 20 个白名单规则，Ant 匹配平均需要 100μs：

| 过滤器                        | 匹配次数  | 单次耗时  | 总耗时       |
|----------------------------|-------|-------|-----------|
| PathPrefixFilter           | 1     | 100μs | 100μs     |
| OAuth2AuthenticationFilter | 1     | 100μs | 100μs     |
| OAuth2AuthorizationFilter  | 1     | 100μs | 100μs     |
| ReplayProtectionFilter     | 1     | 50μs  | 50μs      |
| **合计**                     | **4** | -     | **350μs** |

对于 10,000 QPS 的系统，每天浪费的时间：

- 350μs × 10,000 × 86,400 = **3,024 秒/天 ≈ 50 分钟/天**

## 二、优化方案

### 2.1 核心思想

**"一次检查，多处复用"**

```

    ↓
┌─────────────────────────────────────────────────────────────┐

│   - O(1) 直接读取                                           │
└─────────────────────────────────────────────────────────────┘
    ↓
```

## 三、实现细节

### 3.1 核心组件

```
com.carlos.gateway.whitelist/
├── UnifiedWhitelistProperties.java    # 统一白名单配置
├── UnifiedWhitelistFilter.java        # 统一白名单检查过滤器
├── WhitelistCheckResult.java          # 检查结果封装
└── WhitelistContext.java              # 上下文工具类
```

### 3.2 配置类 (UnifiedWhitelistProperties)

支持分层白名单配置：

```yaml
carlos:
  gateway:
    whitelist:
      enabled: true                    # 是否启用统一白名单
      global-paths:                    # 全局白名单（适用于所有过滤器）
        - /actuator/**
        - /health/**
      auth-paths:                      # 认证白名单（OAuth2 跳过）
        - /oauth2/**
        - /login/**
      waf-paths:                       # WAF 白名单（安全防护跳过）
        - /public/**
      replay-paths:                    # 防重放白名单
        - /webhook/**
      rate-limit-paths:                # 限流白名单
        - /internal/**
      prefix-strip-paths:              # 路径前缀剥离白名单
        - /swagger-ui/**
```

默认自动继承：

- 所有类型默认继承 `global-paths`
- 内置默认白名单（Swagger、Actuator 等）

### 3.3 过滤器 (UnifiedWhitelistFilter)

**执行顺序**: `HIGHEST_PRECEDENCE + 100`（仅次于 Tracing）

**核心逻辑**:

```java
@Override
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    String path = exchange.getRequest().getURI().getPath();

    // 执行统一白名单检查（带缓存）
    WhitelistCheckResult result = performWhitelistCheck(path);

    // 将结果存入 Exchange Attributes
    WhitelistContext.setCheckResult(exchange, result);

    return chain.filter(exchange);
}
```

**匹配缓存**:

```java
// 路径 + 类型的匹配结果缓存
private final ConcurrentHashMap<String, Boolean> matchCache = new ConcurrentHashMap<>();

private boolean isPathWhitelisted(String path, WhitelistType type, Set<String> whitelist) {
    String cacheKey = path + ":" + type.name();
    Boolean cached = matchCache.get(cacheKey);
    if (cached != null) {
        return cached;  // 缓存命中，O(1) 返回
    }
    // 执行匹配并缓存结果
    boolean matched = PathMatchUtil.antMatchAny(whitelist, path);
    matchCache.put(cacheKey, matched);
    return matched;
}
```

### 3.4 上下文工具类 (WhitelistContext)

提供便捷的 API 供其他过滤器使用：

```java
// 检查特定类型的白名单
boolean isAuthWhitelisted = WhitelistContext.isAuthWhitelisted(exchange);
boolean isWafWhitelisted = WhitelistContext.isWafWhitelisted(exchange);
boolean isReplayWhitelisted = WhitelistContext.isReplayWhitelisted(exchange);

// 检查是否在任意白名单中
boolean isAnyWhitelisted = WhitelistContext.isAnyWhitelisted(exchange);

// 通用检查方法
boolean isWhitelisted = WhitelistContext.isWhitelisted(
    exchange, 
    UnifiedWhitelistFilter.WhitelistType.AUTH
);
```

### 3.5 降级兼容

如果统一白名单未启用或结果不存在，过滤器会降级到本地白名单检查：

```java
private boolean isWhitelisted(ServerWebExchange exchange, String path) {
    // 1. 优先使用统一白名单检查结果
    if (WhitelistContext.isAuthWhitelisted(exchange)) {
        return true;
    }

    // 2. 降级兼容：使用本地白名单
    return PathMatchUtil.antMatchAny(properties.getWhitelist(), path);
}
```

## 十、总结

统一白名单优化方案通过"一次检查，多处复用"的核心思想，解决了网关过滤器重复检查白名单的性能问题：

### 核心收益

1. **性能提升**: 白名单检查耗时降低 **71%**
2. **减少重复**: 4 次检查合并为 1 次
3. **统一配置**: 集中管理所有类型白名单
4. **分级支持**: 支持全局、认证、WAF、防重放等分层白名单
5. **降级兼容**: 统一白名单故障时自动回退到本地检查

### 适用场景

- 网关过滤器数量多（>5 个）
- 白名单规则较多（>20 条）
- 高并发场景（>5000 QPS）
- 需要精细化白名单管理
